# نوا (Nava) — فاز اول (فرانت‌اند)

سرویس استریم موسیقی شبیه Spotify، پروژه‌ی درس برنامه‌سازی وب (دانشگاه صنعتی شریف).
این مخزن **فاز اول** است: کل رابط کاربری به‌صورت **ماک** پیاده شده و داده‌ها در
`localStorage` مرورگر نگه‌داری می‌شوند. فاز دوم (بک‌اند Django) صرفاً لایه‌ی داده را
جایگزین می‌کند؛ کامپوننت‌ها دست‌نخورده می‌مانند.

> چهار نقش کاربری: شنونده، هنرمند، پشتیبان، مدیر سامانه — هر کدام تجربه‌ی مخصوص خود.

## اجرا

```bash
npm install
npm run dev      # http://localhost:3000
npm run build    # بیلد تولیدی (Turbopack)
npm start        # اجرای بیلد (سرویس‌ورکر PWA فقط در این حالت فعال است)
npm test         # اجرای تست‌ها (Vitest)
```

### حساب‌های نمایشی (ورود سریع)

رمز عبور اهمیتی ندارد (auth ماک است). در صفحه‌ی ورود، دکمه‌های «ورود سریع نمایشی»
برای هر نقش وجود دارد:

| ایمیل | نقش | توضیح |
| --- | --- | --- |
| `gold@nava.app` | شنونده | اشتراک طلایی (دارای پلی‌لیست و دسترسی زودهنگام) |
| `silver@nava.app` | شنونده | اشتراک نقره‌ای |
| `basic@nava.app` | شنونده | اشتراک پایه (سقف استریم/پلی‌لیست) |
| `artist@nava.app` | هنرمند | تایید‌شده (پنل مدیریت آثار) |
| `pending@nava.app` | هنرمند | در انتظار تایید |
| `support@nava.app` | پشتیبان | داشبورد تیکت و احراز هویت |
| `admin@nava.app` | مدیر | داشبورد کامل + قیمت‌گذاری و حسابرسی |

> برای بازنشانی داده‌ها به حالت اولیه، `localStorage` را پاک کنید (یا کلید `nava-db`).

## استک

- **Next.js 16** (App Router) + **React 19** + **TypeScript**
- **Tailwind v4** + **shadcn/ui** (سبک base-nova، مبتنی بر Base UI) — کاملاً **RTL**
- فونت **استعداد** (variable) به‌عنوان `--font-sans`
- **Zustand** برای state سراسری (سشن، دیتابیس ماک، پخش‌کننده)
- **react-hook-form + zod** برای فرم‌ها
- **Vitest + Testing Library** برای تست
- نمودارها و کاور آهنگ‌ها بدون وابستگی خارجی (SVG/گرادیان تولیدشده)

## معماری و درز اتصال به فاز ۲ (Phase 2 seam)

نقطه‌ی کلیدی نگهداشت‌پذیری: **منطق ماک در یک لایه متمرکز است** تا فاز دوم به‌سادگی
متصل شود.

- **`lib/stores/db-store.ts`** نقش «دیتابیس» را دارد: کل وضعیت در حافظه + ذخیره در
  `localStorage`. هر تغییر داده (ساخت پلی‌لیست، فالو، تایید هنرمند، انتشار اثر،
  تغییر قیمت‌ها و ...) یک **action** اینجاست. در فاز دوم فقط بدنه‌ی این action‌ها به
  فراخوانی API (Django/DRF) تبدیل می‌شود؛ نام action‌ها و شکل `types.ts` ثابت می‌ماند.
- **`lib/stores/player-store.ts`** یک **transport شبیه‌سازی‌شده** است (صف، تکرار،
  شافل، seek، موقعیت). در فاز ۱ یک تایمر (`PlayerTransport`) موقعیت را جلو می‌برد؛ در
  فاز ۲ کافی است یک `<audio>` واقعی جای تایمر بنشیند و رویدادهای `timeupdate`/`ended`
  را به همان action‌ها بدهد. منطق صف/تکرار/شافل تغییری نمی‌کند.
- **سیاست پخش** (سقف استریم روزانه‌ی اشتراک پایه) در `lib/hooks/use-playback.ts` است،
  جدا از مکانیزم transport، تا استور خالص و قابل‌تست بماند.

## قراردادهای توسعه

- **نام‌گذاری:** کامپوننت‌ها `PascalCase`؛ فایل‌ها و فولدرها `kebab-case`؛ توابع/متغیرها
  `camelCase`؛ تایپ‌ها `PascalCase`. هر store با پیشوند `use` (مثل `useDb`).
- **RTL:** همیشه از کلاس‌های منطقی Tailwind استفاده می‌شود (`ms/me`, `ps/pe`,
  `start/end`, `border-s/e`) نه `left/right`. نوار پیشرفت و کنترل‌های پخش به‌عمد
  `dir="ltr"` هستند تا مثل هر پخش‌کننده‌ای خوانده شوند.
- **Server/Client:** layoutهای ریشه Server Component‌اند؛ صفحات تعاملی `"use client"`.
  مسیرهای داینامیک با `useParams()` خوانده می‌شوند (سازگار با async params در Next 16).
- **اعداد/تاریخ:** همیشه از `lib/format.ts` (ارقام و تقویم فارسی).
- **Base UI:** به‌جای `asChild` از پراپ `render={<...>}` استفاده می‌شود؛ Sliderها مقدار
  آرایه‌ای می‌گیرند.
